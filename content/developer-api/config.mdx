# Config — built-in integrations

Some third-party integrations ship with LACORE and are wired up purely through config — no code, no
core edits. They live in `configs/cfg-integrations-sh.lua`, gated so they only turn on when the target
resource is actually running.

## Dispatch systems

LACORE forwards its automatic calls (911 / 311 / Panic / Crime Broadcast / Shot Spotter / Requesting
LEO·Fire-EMS·Coroner / Tow / Created Incident) into your dispatch script. A pluggable adapter subscribes
to [`lacore:api:callCreated`](/developer-api/events) and routes each call to the active dispatch — LACORE's
own dispatch/MDT keeps working either way.

| Dispatch | Resource | How it's sent |
|----------|----------|---------------|
| **tk_dispatch** | `tk_dispatch` | `addCall` export + `setCallsign` (callsign sync) |
| **ps-dispatch** | `ps-dispatch` | `CustomAlert` (client export, relayed) |
| **cd_dispatch** | `cd_dispatch` | `cd_dispatch:AddNotification` event |
| **core_dispatch** | `core_dispatch` | `core_dispatch:addCall` event |
| **rcore_dispatch** | `rcore_dispatch` | `rcore_dispatch:server:sendAlert` event |
| **linden_outlawalert** | `linden_outlawalert` | `wf-alerts:svNotify` event |

```lua
-- configs/cfg-integrations-sh.lua

-- Pick your dispatch: "auto" uses the first supported one that's running.
Integrations.dispatch = "auto"   -- "auto" | "ps-dispatch" | "cd_dispatch" | "tk_dispatch" | … | false

Integrations.dispatchJobs = { "police" }        -- fallback recipients (framework jobs, ESX/QB)
Integrations.dispatchForwardUnmapped = false    -- forward types not listed below?
Integrations.dispatchSyncCallsigns = true       -- push callsigns where supported (tk_dispatch)

-- ONE neutral map for every dispatch — each adapter translates it to its own fields.
Integrations.dispatchCalls = {
    ["911 Emergency"]       = { code = "911",   priority = "high",   jobs = { "police", "ambulance" }, blip = { sprite = 280, color = 1 } },
    ["Panic Button"]        = { code = "10-99", priority = "high",   jobs = { "police" }, flash = true },
    ["Shot Spotter"]        = { code = "shots", priority = "high",   jobs = { "police" }, flash = true },
    ["Crime Broadcast"]     = { code = "crime", priority = "high",   jobs = { "police" } },
    ["Requesting Fire/EMS"] = { code = "10-52", priority = "high",   jobs = { "ambulance" } },
    -- … 311 / Requesting LEO / Coroner / Tow / Created Incident
}
```

Each entry is a **neutral** shape — `code` (10-code), `priority` (`"low"`/`"medium"`/`"high"`), `jobs`
(recipients), `blip` (`{ sprite, color, scale }`) and `flash`. The active adapter converts it to that
dispatch's own field names, so you configure it **once** no matter which dispatch you run. Coords,
message and location come from the LACORE call.

<Callout type="warning">
  **Set `jobs` to your framework's job names** (`police`, `ambulance`, …) — these are ESX/QB jobs, not
  LACORE departments. With the wrong names, the dispatch shows the call to nobody.
</Callout>

<Callout type="info">
  Running a dispatch that isn't listed? You don't edit LACORE — write a tiny external resource that
  subscribes to `lacore:api:callCreated` and calls your dispatch's export. Copy the
  [example resource](/developer-api/examples) and point it at your target.
</Callout>
