# Developer API

A small, stable **exports + events** surface so other resources can talk to LACORE / LACORE MDT.
Every export wraps an existing internal function — no new backend logic, no schema of its own. Feed
dispatch calls in, read officer / duty state, look records up, issue BOLOs, react to CAD events, and
open the MDT from your own scripts — **without ever editing the (encrypted) core**.

<Callout type="info">
  **Resource name.** Exports are called by the *running* resource's name — `exports['lacore']` for the
  full core, or `exports['lacore-mdt']` for the standalone MDT product. The code is identical; only the
  prefix differs. Examples use `lacore`.
</Callout>

Check the shipped version at runtime:

```lua
local version = exports['lacore']:GetApiVersion()   -- e.g. "1.0.0"
```

## The pattern

Two ways to integrate — both from your **own** resource, never touching LACORE's files:

- **Consume** — `AddEventHandler('lacore:api:*')` to react to dispatch calls, duty / status changes,
  reports, charges and BOLOs. See [Events](/developer-api/events).
- **Drive** — `exports['lacore']:CreateCall{…}` / `ResolveCall(…)` / `QueryPerson(…)` … to push data
  into LACORE. See [Server](/developer-api/server).

<Callout type="warning">
  **Trust model.** Server exports are meant to be called by other **server** resources (trusted).
  Client exports only open / close the local UI and can't grant authority — call creation, resolution
  and BOLO issuing stay server-authoritative.
</Callout>

- **Stability.** Export names and payload shapes are the contract. New fields may be added; existing
  ones won't be renamed within a major version (`GetApiVersion`).
- **Alongside the old exports.** The existing standalone exports (`isRolePresent`, `getUserRoles`,
  `getUserData`, `IsClientVerified`, `IsGloballyBanned`, `GetFramework`, …) keep working unchanged —
  this API sits next to them.
