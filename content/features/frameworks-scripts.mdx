# Frameworks & Scripts

LACORE is **standalone-first** — it ships with its own identity, jobs and duty system and needs no
framework. But when you already run one, or use popular paid resources, LACORE **integrates with them
automatically**. Everything below is **auto-detected**: if the resource is running, the integration
turns on; if it isn't, LACORE simply carries on standalone. No integration is ever required.

## Supported frameworks

| Framework | Resource | Status |
| --- | --- | --- |
| **Standalone** | — (LACORE's own identity/multichar) | ✅ Default |
| **ESX** | `es_extended` | ✅ Supported |
| **QBCore** | `qb-core` | ✅ Supported |
| **QBox** | `qbx_core` | ✅ Supported |

When a framework is detected, LACORE pulls **jobs and identity** from it:

- **Framework job = duty authorisation.** A player whose framework job maps to an agency can go on
  duty in the MDT / dispatch **without also needing a LACORE Discord duty role** — verified
  server-side, so it can't be spoofed.
- **Instant job sync.** Job changes apply immediately via `esx:setJob` / `esx:playerLoaded` (and the
  QBCore equivalents), not a slow poll.
- **Identity → RP name.** The framework character name is adopted as the player's LACORE RP name
  (unless they've set one via `/char` or multichar, which always wins).
- **Optional auto-duty.** `Bridge.autoDuty` can put mapped jobs on/off duty the moment the framework
  job changes.

Configure the job → agency map in `configs/cfg-bridge-sh.lua`. See the
[Framework Bridge](/features/bridge) page for the full API and mapping.

<Callout type="warning">
  **Load order:** `ensure` your framework **before** `lacore` in `server.cfg` (detection runs at
  startup). LACORE keeps no hard dependency, so it still boots standalone if none is present.
</Callout>

## Supported scripts

LACORE plugs into these third-party resources when they're installed — configured in
`configs/cfg-integrations-sh.lua` (each toggle is `"auto"` = on when present, or `false` to disable).

| Script | What LACORE does with it | How |
| --- | --- | --- |
| **vms_housing** | A queried person's **registered property addresses** show in the MDT person record. | Native export (`GetPlayerProperties`) |
| **vms_identity** | Character **identity (name)** is read automatically for MDT records. | Via the framework bridge |
| **vms_multichars** | The **active character** (identity + identifier) is recognised; querying an online player with no LACORE profile still returns a record. | Via the framework bridge |

<Callout type="info">
  `vms_identity` and `vms_multichars` are framework resources (ESX / QBCore / QBox), so they work
  through LACORE's [framework bridge](/features/bridge) — install one of the supported frameworks and
  they're picked up automatically.
</Callout>

<Callout type="info">
  Want another script supported? Ask in our [Discord](https://discord.gg/b6YbvVEvYp) — if it exposes
  a clean export or event, it can usually be bridged the same way.
</Callout>
