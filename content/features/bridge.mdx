# Framework Bridge

LACORE runs standalone with its own identity layer, but it can also **bridge to an existing
framework** so jobs and identity come from ESX, QBCore or QBox.

## Auto-detection

At load, the bridge (`modules/bridge/`) detects the active framework and exposes a uniform API:

| Function | Returns |
| --- | --- |
| `Bridge.GetJob(src)` | The player's job. |
| `Bridge.GetName(src)` | The player's character name. |
| `Bridge.GetIdentifier(src)` | A stable identifier. |
| `Bridge.IsLeo(src)` / `Bridge.IsEms(src)` | Duty checks. |
| `Bridge.Notify(src, msg)` | Framework-native notification. |

## Job → department mapping

Map your framework's job names to LACORE departments in `configs/cfg-bridge-sh.lua`:

```lua
Bridge.jobMap = {
  ['police']    = { dept = 'lapd', job = 'Law Enforcement' },
  ['sheriff']   = { dept = 'lasd', job = 'Law Enforcement' },
  ['ambulance'] = { dept = 'ems',  job = 'AMR' },   -- EMS/Fire in-game job string
  -- ...
}

Bridge.autoDuty = false  -- optionally auto-set duty from the framework job
```

A framework job that maps to an agency counts as **duty authorisation** — so an ESX/QB officer can go
on duty without also holding a LACORE Discord duty role. See
[Frameworks & Scripts](/frameworks-scripts) for the full compatibility list.

## ESX characters as the character source

On an **ESX** server you don't want players filling out LACORE's `/char` on top of their existing ESX
identity. With `Bridge.useFrameworkCharacters = true` (the default), LACORE reads each player's ESX
character straight from the database on load and uses it everywhere — the MDT/CAD person lookup,
`/profile`, and the on-screen name.

| Pulled from ESX | Source |
| --- | --- |
| Name (first + last), date of birth, sex, height, phone number, job | `users` row for the current identifier |
| Registered vehicles (plates in the CAD query) | `owned_vehicles` for that owner |

It works with **esx_multicharacter** automatically: each ESX character is its own `users` row, so LACORE
keeps one character per ESX character and re-syncs on every switch.

<Callout type="warning">
  While ESX drives the character, its **identity is read-only in LACORE** (name / DOB / sex / height /
  phone / job) — ESX's database stays the single source of truth. Records, notes and relationships stay
  editable in LACORE. Requires **oxmysql**. Set `Bridge.useFrameworkCharacters = false` to keep LACORE's
  own `/char` identity even while ESX still handles jobs.
</Callout>

## Anticheat interop

The bridge marks legitimate framework events so LACORE's event-honeypot anticheat never flags them
(`Bridge.IsFrameworkEvent`).

<Callout type="info">
  No framework? LACORE uses its own multi-character identity — nothing else to configure.
</Callout>
