# Frameworks & Scripts

LACORE is **standalone-first** — it ships with its own identity, jobs and duty system and needs no
framework. But when you already run one, or use popular paid resources, LACORE **integrates with them
automatically**: if the resource is running, the integration turns on; if not, LACORE carries on
standalone. Nothing here is ever required.

## Supported frameworks

<FrameworkGrid items={frameworks} />

When a framework is detected, LACORE pulls **jobs and identity** from it:

- **Framework job = duty authorisation.** A player whose framework job maps to an agency can go on
  duty in the MDT / dispatch **without also needing a LACORE Discord duty role** — verified
  server-side, so it can't be spoofed.
- **Instant job sync.** Job changes apply immediately via `esx:setJob` / `esx:playerLoaded` (and the
  QBCore equivalents), not a slow poll.
- **Identity → RP name.** The framework character name becomes the player's LACORE RP name (unless
  they set one via `/char` or multichar, which always wins).
- **Optional auto-duty.** `Bridge.autoDuty` can put mapped jobs on/off duty the moment the framework
  job changes.

Map your framework jobs to agencies in `configs/cfg-bridge-sh.lua` — full API on the
[Framework Bridge](/features/bridge) page.

<Callout type="warning">
  **Load order:** `ensure` your framework **before** `lacore` in `server.cfg` (detection runs at
  startup). LACORE keeps no hard dependency, so it still boots standalone if none is present.
</Callout>

## Supported scripts

Third-party resources LACORE plugs into when installed — configured in
`configs/cfg-integrations-sh.lua` (each toggle `"auto"` = on when present, or `false` to disable).

<Callout type="info">
  `vms_identity` and `vms_multichars` are framework resources (ESX / QBCore / QBox), so they work
  through LACORE's [framework bridge](/features/bridge) — install a supported framework and they're
  picked up automatically.
</Callout>

  a clean export or event, it can usually be bridged the same way.
