# Discord & Duty — `configs/cfg-server-sv.lua`

Server-only config (never sent to clients). Handles dev mode, the built-in Discord role
authentication, the on-duty whitelist, and the webhook log channels.

## Dev mode

```lua
devmode = GetConvar("lacore_devmode", "false") == "true"
```

See the [`lacore_devmode` convar](/configuration/convars#dev-mode). When on, it disables webhooks,
bypasses **all** connect checks, and grants every player full staff/dev permissions. Never on in
production.

## Access control (whitelist & Discord on/off)

Framework servers (ESX / QBCore / QBox) usually handle identity and permissions themselves and don't
need LACORE's Discord layer or its members-only whitelist. Switch either off here:

```lua
AccessControl = {
    whitelist    = true,    -- members-only join gate (/wl). false = disabled entirely.
    discordRoles = true,    -- require + read Discord roles and bridge them to ACE. false = off.
    membership   = "auto",  -- who counts as a "member" for member-only perks. See below.
}
```

| Toggle | `false` means |
| --- | --- |
| `whitelist` | The members-only gate is off: `/wl` does nothing and no one is ever rejected with “members only”. **Bans and the global-ban network still apply.** |
| `discordRoles` | LACORE no longer requires Discord at connect (no “join our Discord” rejection), doesn't read Discord roles, and doesn't bridge them to ACE. Duty-role gating (`DutyRoles`) is skipped, and Staff/Dev powers come from **ACE / txAdmin** only. |

### Membership (member-only perks)

`membership` decides who counts as a **member** for member-only perks — going on agency duty (`/onduty`),
spawning member vehicles, and voting on the community board. This is separate from the join whitelist;
membership normally rides on the Discord **“Member”** role.

| Value | Behaviour |
| --- | --- |
| `"auto"` *(default)* | With Discord role auth **on**, a member is anyone holding the Discord “Member” role. With Discord auth **off** there is no role to read, so **everyone is a member**. |
| `"open"` | Everyone is always a member, even with Discord auth on. |
| `"role"` | Always require the Discord “Member” role — locks the perks for everyone when Discord auth is off, so only use it with Discord on. |

<Callout type="info">
  On an ESX / QBCore / plug-and-play server (`discordRoles = false`), `"auto"` is what makes `/onduty`
  and the other member-only perks **just work** — otherwise, with no “Member” role to read, they would be
  silently locked for everyone. Duty is still authorised by the framework job (see the
  [Framework Bridge](/features/bridge)); `membership` only governs LACORE's own member gate.
</Callout>

<Callout type="info">
  **Plug-and-play:** even with `discordRoles = true`, Discord is only *enforced* once it's actually
  configured (a bot token **and** guild id are set). On a fresh install with no token, Discord stays
  **idle** — players connect without it and no roles are read — and the moment you set the convars, role
  features light up. So you never have to touch this to get the core running.
</Callout>

<Callout type="warning">
  With `discordRoles = false` (or while Discord is idle), admin powers don't come from Discord — grant
  them through ACE (e.g. `add_ace group.admin lacore allow` in `server.cfg`) or txAdmin, and duty
  authorisation comes from the framework job (see the [Framework Bridge](/features/bridge)).
</Callout>

## Discord role authentication (built-in)

LACORE reads Discord roles directly via a bot token (no external `discordroles` resource). Roles are
cached per player and exposed via `GetDiscordRoles()` / `IsDiscordRolePresent()`.

```lua
DiscordAuth = {
    token        = GetConvar("lacore_discord_token", ""),
    guild        = GetConvar("lacore_discord_guild", ""),
    cacheSeconds = 300,   -- how long fetched roles are cached

    -- Map role NAMES (used in code) → your Discord role IDs.
    -- Enable Developer Mode in Discord, right-click a role → Copy Role ID.
    roles = {
        Member        = "1271895547485356203",
        Staff         = "1234427956432994321",
        Dev           = "1280438326582837278",
        LeadDeveloper = "1395534398299832431",
        LOA           = "1310081058762653696",
        LE            = "1285460493619953720",
        -- Supporter / Donator tiers … (leave empty = "never present")
    },
}
```

<Callout type="info">
  The bot must be **in your guild** and have the **Server Members Intent** enabled (Developer Portal
  → Bot → Privileged Gateway Intents). Leave a role ID empty to treat it as never present.
</Callout>

Role names referenced elsewhere (profile badges, Big Brother access, duty whitelist) must match keys
in this `roles` table.

## Duty whitelist

Closes the gap where any client could self-assign a duty job. Checked server-side in `ondutyServer`.

```lua
DutyRoles = {
    ["Law Enforcement"] = "LE",  -- requires the LE role
    ["Fire/EMS"]        = "",    -- open by default
    ["Coroner"]         = "",    -- open by default
}
```

- `devmode` and `Staff` / `Dev` bypass the check.
- An empty value (`""`) means no restriction — anyone may take that job.
- Values are role names from `DiscordAuth.roles`.

## Webhooks

Webhook URLs are read from convars so they never live in the code. Any left empty disables that
channel.

```lua
Webhooks = {
    ServerLog       = GetConvar("lacore_webhook_serverlog", ""),
    FiveMLiveChat   = GetConvar("lacore_webhook_livechat", ""),
    AdminLog        = GetConvar("lacore_webhook_adminlog", ""),
    DispatchLog     = GetConvar("lacore_webhook_dispatch", ""),
    UserReports     = GetConvar("lacore_webhook_reports", ""),
    PrisonLog       = GetConvar("lacore_webhook_prison", ""),
    BleeterLog      = GetConvar("lacore_webhook_bleeter", ""),
    txAdminAdminLog = GetConvar("lacore_webhook_txadmin", ""),
}
```

Set the matching convars in `server.cfg` — see [Server Convars](/configuration/convars#discord-webhooks).

## Discord Rich Presence — `configs/cfg-presence-sh.lua`

Shows a live status on each player's Discord profile with clickable buttons, so their friends can see the
server and join. Set the fields directly in the config — a plain `set` convar is server-only and can't
reach the client, so presence is **not** driven by convars.

```lua
RichPresence = {
    enabled    = true,
    appId      = 0,              -- Discord Application ID (number); 0 = disabled
    largeAsset = "icon",         -- art asset key uploaded in the Dev Portal
    smallAsset = "",
    discordInvite = "discord.gg/YOURINVITE",   -- "Discord" button
    website       = "",                        -- "Website" button (blank = hidden)
    lines = {                    -- status text; {players} / {aop} / {discord} are filled live
        "{players} Players | AOP: {aop}",
        "{players} Players | {discord}",
    },
    rotateSeconds = 4,           -- seconds between rotating through the lines above
    connectCard   = false,       -- true = a "Join Discord" card on the connect screen; false = instant connect
}
```
